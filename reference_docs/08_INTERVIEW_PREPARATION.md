# 🎤 08. 면접 대비 질의응답 (Interview Preparation Q&A)

본 문서는 Security SOAR 프레임워크를 직접 설계하고 개발한 **아키텍트/리드 개발자** 포지션으로 기술 면접(Technical Interview)에 임할 때 예상되는 심층 질문과 모범 답변(Best Practice), 그리고 면접관이 파고들 수 있는 **꼬리 질문(Follow-up Questions)** 들을 정리한 문서입니다.

단순한 기능 설명을 넘어, "왜 이 구조를 선택했는지(Why)", "발생할 수 있는 트레이드오프(Trade-off)는 무엇인지"에 대한 철학적인 방어 논리를 포함합니다.

---

## 🏗️ 1. 아키텍처 패턴 및 설계 철학

### 📊 Q1. "SOAR 프레임워크의 코어 아키텍처를 설계할 때 가장 중요하게 생각한 디자인 패턴은 무엇이고, 왜 선택했나요?"
**A.** 가장 중요하게 생각한 것은 **팩토리 패턴(Factory Pattern)**과 **전략 패턴(Strategy Pattern)**의 결합이었습니다. 보안 도메인의 특성상 "새로운 위협(Event)"과 "새로운 조치(Action)"가 계속해서 추가되기 때문입니다.
`SecurityActionFactory`를 통해 조치 로직의 생성을 캡슐화하고, 각 조치들은 `ISecurityAction` 인터페이스를 구현(Strategy)하도록 강제했습니다. 이로 인해 코어 엔진(`SecurityAlertHandler`)은 구체적으로 어떤 조치가 실행되는지 알 필요 없이, 다형성(Polymorphism)을 활용해 `action.execute()`만 호출하면 되는 극단적으로 낮은 결합도(Loose Coupling)를 확보할 수 있었습니다.

#### 🎯 [꼬리 질문] "팩토리 패턴을 썼다고 하셨는데, 새로운 Action이 추가될 때 팩토리 클래스 본문의 `switch-case` 문을 계속 수정해야 한다면 그건 완벽한 OCP(개방-폐쇄 원칙) 위배 아닌가요?"
**A. (방어 논리)** 맞습니다. 현재 초기 버전에서는 직관성을 위해 `switch` 문으로 구현되어 있어 엄밀히 말해 OCP를 완벽히 충족하지는 못합니다. 하지만 이 기술 부채(Technical Debt)를 이미 인지하고 있으며, 차기 리팩토링 단계에서 **단일 책임 원칙(SRP)**과 결합하여 커스텀 메타데이터와 **Reflection API(`Type.forName()`)**를 활용하는 동적 팩토리 구조로 전환할 계획을 설계 문서(`05_REFACTORING_OPPORTUNITIES.md`)에 명시해 두었습니다. 이렇게 되면 Apex 코드 수정 없이 메타데이터 설정만으로 팩토리가 영구적으로 동작하게 됩니다.

### 📊 Q2. "플랫폼 이벤트(Platform Event)를 비동기 메세지 버스로 활용했는데, 기존의 @future나 Queueable 대신 굳이 플랫폼 이벤트를 쓴 아키텍처적 이유가 있나요?"
**A.** 보안 트랜잭션의 **안정성 확보** 와 **디커플링(Decoupling)** 때문입니다. TxnSecurity 트리거는 사용자가 리포트를 열거나 로그인하는 가장 핵심적인 실시간 경로에 개입합니다. 만약 여기서 무거운 동기/비동기 로직이나 외부 Callout 로직이 조금이라도 지연을 일으키면 시스템 전체 장애로 이어질 수 있습니다.
플랫폼 이벤트(`SecurityAlert__e`)를 Publish 후 곧바로 트랜잭션을 종료시킴으로써(Fire and Forget), Salesforce 코어 트랜잭션과 보안 프레임워크의 실행 컨텍스트를 물리적으로 완전히 격리시켰습니다.

#### 🎯 [꼬리 질문] "플랫폼 이벤트 제약(Limit)이 있는데 하루에 수십만 건의 위협 이벤트가 쏟아지면 배달이 누락되거나 제한에 걸리지 않나요? 어떻게 방어했습니까?"
**A. (방어 논리)** 두 가지 레이어로 방어했습니다.
첫째, 트리거 앞단에서 동작하는 `SecurityGuard`의 **캐싱 메모리 로직**과 `SecurityActionThrottle`의 **서킷 브레이커**를 통해 의미 없는 중복 이벤트가 큐에 쌓이는 것을 1차로 잘라냈습니다. (예: 동일 IP의 반복 침해는 분당 1회만 허용)
둘째, 만약 정말로 트래픽이 몰릴 경우 `SecurityAlertHandler`가 플랫폼 이벤트를 수신한 직후, 즉시 처리하기 버거운 작업은 팩토리 패턴을 이용해 또다시 **청크(Chunk) 단위의 Batch 나 Queueable 로 위임(`InterfaceBridge`)** 하도록 2차 방어선을 구축했습니다.

## ⚡ 2. 트랜잭션, 성능 및 리미트 (Salesforce Limits)

### 📊 Q3. "프레임워크 내부에서 정책(Policy)과 설정(Action Mapping)을 판단하기 위해 자주 DML(SOQL) 조회를 하는 것 같은데, SOQL Limits 에 대한 최적화는 어떻게 했나요?"
**A.** SOAR 프레임워크의 설정값들은 모두 일반 SObject(Custom Object)가 아닌 **Custom Metadata Type(`__mdt`)**으로 설계했습니다. 커스텀 메타데이터 조회는 Salesforce 일반 SOQL 100 Limit 카운트에 포함되지 않으므로 쿼리 횟수 제한으로부터 안전합니다.
또한, 트랜잭션 내에서 설정이 반복 호출되는 것을 방지하기 위해 `SecurityGuard` 유틸리티 내에 `static Map` 변수를 활용하여 한 번 읽어온 정책 메타데이터를 **Static Variable Memory Cache**에 올려 로드 타임을 O(1)로 최적화했습니다.

#### 🎯 [꼬리 질문] "Static 변수에 캐시한다고 하셨는데, 트랜잭션이 길어지거나 다른 사용자의 컨텍스트가 섞이는 비동기 로직에서는 해당 캐시 메모리가 풀려서 오히려 독이 될 수도 있지 않나요?"
**A. (방어 논리)** 정확한 지적입니다. Salesforce의 Static 변수는 "단일 Execution Context" 내에서만 유지됩니다. 동기 트리거 단계에서는 매우 효율적이지만, 비동기(Queueable/Batch)로 넘어가는 순간 컨텍스트가 초기화되므로 어차피 다시 조회해야 합니다. 
이를 보완하기 위해 향후 프레임워크 리팩토링 포인트로 **Singleton/Registry 패턴** 계층을 하나 더 두어, Execution Context에 무관하게 플랫폼 캐시(Platform Cache - Org Cache)까지 붙이는 중앙 관리자 래퍼(Wrapper) 층을 도입하는 설계(`05` 문서 5번 항목)를 고민하고 적용 대기 중에 있습니다.

### 📊 Q4. "수많은 액션들을 동시에 처리하다 보면 `System.LimitException: Too many callouts: 101` 에러가 발생할 위험이 커 보이는데요."
**A.** Callout 100 Limits 방어를 위해 프레임워크 아키텍처 설계 단계부터 **인터페이스 실행 체계의 분리(`SecurityInterfaceBridge`)**를 기획했습니다.
외부로 데이터를 쏘는 모든 연동 로직은 반드시 메타데이터(`SecurityIntegration__mdt`)에 의해 실행 모드(`REALTIME`, `QUEUEABLE`, `BATCH`)가 강제되도록 했습니다. 하나의 트랜잭션에서 여러 Callout이 병렬로 요구될 경우, 브릿지가 이를 인지하고 트랜잭션 맥락에 맞춰 `System.enqueueJob` 또는 `Database.executeBatch` 로 비동기 위임함으로써 Limits를 회피합니다.

#### 🎯 [꼬리 질문] "그렇다면 Queueable로 보냈을 때, 또 Queueable 안에서 또 Queueable을 부르는 Depth 제한(Chaining Limits)에 걸릴 위험은요?"
**A. (방어 논리)** 이를 막기 위해 `SecurityActionExecutor` 레벨에서 영리한 큐잉을 구현할 계획입니다(설계 문서 참조). 낱개의 큐를 계속 Chain 거는 것이 아니라, 수행해야 할 Callout 작업들을 List 구조 담아 하나의 `BulkAsyncActionJob(Queueable)` 로 던져서, 1개의 비동기 트랜잭션 안에서 For 루프를 돌며 여러 번의 Callout을 일괄 처리(벌크 처리)하게 함으로써 Chaining Depth 한도를 소모하지 않도록 설계했습니다.

## 🔒 3. 인바운드 보안 및 해킹 방어

### 📊 Q5. "외부 메신저(Slack/Teams) 알림의 버튼을 누르면 인바운드 웹훅을 통해 Salesforce 내부 계정이 락(Lock) 된다고 하셨는데, 그 웹훅 URL이 유출되면 누구나 공격/조작할 수 있는 것 아닌가요?"
**A.** 가장 고민했던 보안 취약점입니다. 오픈된 웹훅을 방어하기 위해 **다중 인증 및 무결성 검증 아키텍처(`SecurityActionInboundHandler`)**를 구축했습니다.
1. **토큰 검증**: 인바운드 요청 URL에 1회성 JTI(JWT Request ID) 혹은 암호화된 토큰을 심어 보냅니다.
2. **Replay Attack 방어**: 한 번 사용된 토큰 키 조합은 즉시 소모(Mark as Used) 처리되어, 해커가 패킷을 가로채서 똑같은 URL을 다시 호출해도 로직이 거부합니다.
3. **Payload 무결성**: 전달된 데이터가 조작되지 않았는지 (예: A계정을 막으라고 보냈는데 해커가 B계정으로 파라미터를 변조) HMAC SHA256 서명 기반 검증을 추가로 적용할 수 있도록 확장 포인트를 만들었습니다.

#### 🎯 [꼬리 질문] "인바운드 로직 처리를 Apex REST가 아닌 Site(커뮤니티/Guest User) 쪽으로 열었다면, Guest User가 OWD나 데이터 보안 제약 때문에 유저 계정을 동결시킬 권한이 없을 텐데 어떻게 해결했습니까?"
**A. (방어 논리)** 맞습니다. Guest User는 구조적으로 `User` 오브젝트의 민감한 필드(IsActive 등)를 업데이트할 권한이 없습니다. 이를 해결하기 위해 **권한 이관(Privilege Escalation) 래퍼(Wrapper)** 클래스를 도입했습니다.
`SecurityPrivilegeEscalator`와 같이 `without sharing` 키워드가 선언된 아주 좁은 영역의 브릿지 클래스를 만들고, 인바운드 핸들러가 토큰/보안 검증을 "완벽히 통과했을 때만" 이 래퍼 클래스로 넘겨 시스템 관리자 권한(명시적 DML 수행 권한)으로 해당 비즈니스 로직을 대리 실행하도록 설계했습니다. 최소 권한의 원칙(POLP)을 지키면서도 우회 경로를 안전하게 확보한 것입니다.

## 💡 4. 확장성과 유지보수 경험치 (Seniority)

### 📊 Q6. "1년 뒤에 이 프레임워크를 다른 개발자가 인수인계받는다면 가장 어려워할 부분은 어디일까요? 그리고 어떻게 대비해 두셨나요?"
**A.** 가장 큰 허들은 `SecurityGuard`나 `SecurityAlertHandler` 와 같이 수많은 책임이 모여있는 **코어 유틸리티 클래스의 파악**일 것입니다. 초기 개발 단계에서는 속도와 응집력을 위해 여러 기능(정책 읽기, 임계치 계산, 권한 이관 등)을 뭉쳐 놓았기 때문입니다.
이를 대비해 **2가지 방어책**을 마련했습니다.
1. **철저한 문서화 아키텍처**: 단순히 코드 주석이 아니라, 기획 배경(01) -> 아키텍처(02) -> 클래스 명세(03) -> 리팩토링 제안(05) -> 외부 설정 구조(06)로 이어지는 **마크다운(MD) 기반의 세계관**을 구축해 배포했습니다.
2. **기술 부채 문서화**: `05_REFACTORING_OPPORTUNITIES.md` 라는 문서를 통해 "현재 이 코드는 완벽하지 않으며, 단일 책임 원칙(SRP) 위배를 막기 위해 향후 `SecurityUserHelper`, `SecurityTimeHelper` 등으로 쪼개(Facade 패턴 도입)야 한다"는 가이드라인을 남겨두어, 후임자가 망설임 없이 구조를 개선할 수 있는 명분을 제공했습니다.

#### 🎯 [꼬리 질문] "본인이 이 프레임워크를 처음부터 다시 만든다면 가장 먼저 스크래치(Scratch)부터 다르게 설계할 부분이 있다면 무엇입니까?"
**A. (통찰력 어필)** 만약 0에서 다시 시작한다면 **이벤트 소싱(Event Sourcing) 기반의 상태 머신(State Machine)** 구조를 도입해 볼 것 같습니다.
현재는 한 번의 위협(이벤트)이 들어오면 `Handler`가 바로 `Action`으로 일직선으로 처리해 버립니다. 하지만 보안 위협은 "경고 -> 1시간 지켜봄 -> 격상 -> 락다운"과 같이 시간의 흐름에 따른 '상태 변화'가 중요합니다. 나중에 다시 만든다면 무거운 오케스트레이터를 만드는 대신, 위협 데이터를 일차원적 로그가 아닌 '이벤트 스트림'으로 기록하고, 각 액션 모듈이 이 상태 머신의 스냅샷을 구독(Observe)하며 독립적으로 움직이는 **비동기 마이크로서비스 형태의 Apex 프레임워크**를 실험해 보고 싶습니다.
